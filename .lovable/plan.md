## Root cause

The latest Talking Video (`1171b5a4…`, kling task `885322483212095511`) finished with the base Kling video saved as `video_url` and metadata still on `stage: "base_video"` — `lipsync_task_id` is `null` and `silent_fallback` is `false`. That means the lip-sync stage 2 (which would have re-uploaded the ElevenLabs audio onto the clip) was never even attempted.

The culprit is the `recover` action in `supabase/functions/generate-video/index.ts` (around lines 587–671). When the Video Hub / client calls it, the function:

1. Selects every `generated_videos` row for the user with `status='processing'` and a `kling_task_id` — **including talking-video rows**.
2. Hits `image2video/{taskId}` (which works because stage 1 is also an image2video task).
3. Sees `task_status='succeed'`, downloads the base clip, and writes `status='complete'` + `video_url=<base mp4>` directly to the row.

That short-circuits `poll-stuck-videos`, which is the only function that knows about the two-stage talking pipeline (base → lip-sync). Once the row flips to `complete`, `poll-stuck-videos` filters it out (`.eq('status','processing')`), so lip-sync is never submitted and the user gets a silent clip with the right lip motion that was generated by Kling's own prompt — not by the ElevenLabs audio.

## Fix

Make the `recover` action in `generate-video` ignore talking-video rows and let `poll-stuck-videos` own them end-to-end.

### Changes (single file: `supabase/functions/generate-video/index.ts`)

1. **Filter the recovery query** to exclude talking-video rows:
   ```ts
   .from("generated_videos")
   .select("kling_task_id, model_name, workflow_type")
   .eq("user_id", userId)
   .eq("status", "processing")
   .not("kling_task_id", "is", null)
   .neq("workflow_type", "talking_video")
   ```

2. **Defensive skip inside the loop** in case future code paths add rows without going through that filter:
   ```ts
   if (video.workflow_type === "talking_video") continue;
   ```

3. **Same guard on the `stuckQueueJobs` block** below (lines 674–): only resolve queue rows whose job_type is `video` / `video_multishot` (already filtered), but add a comment so this isn't extended to `talking_video` later.

### Repair the orphaned row (one-time)

The current broken row (`1171b5a4-a6e3-4c70-8e49-053d833d4371`) is already `complete` and points at the silent base video. To recover it we can either:

- **Option A (recommended):** Reset that single row back to `processing` with `metadata.stage='base_video'`, clear `video_url`, and the next `poll-stuck-videos` tick (runs ~minute-ly) will pick it up, re-fetch the Kling base, sign the existing ElevenLabs audio at `fe45fd27…/talking/10a8dc52…mp3` (still in `generated-audio` bucket), submit lip-sync, and finish properly. Also reset its queue job (`10a8dc52…`) back to `processing` so bookkeeping stays consistent.
- **Option B:** Leave the row and just tell the user to regenerate.

I'll go with Option A in a one-off migration / SQL update so the user gets the audio version of the video they already paid for, then deploy the code fix so it never happens again.

### Out of scope

- No UI changes.
- No changes to `poll-stuck-videos` (it's already correct).
- No changes to credits, voice picker, or ElevenLabs settings.
